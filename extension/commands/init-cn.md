# 棕地專案 SDD 引導系統 (Brownfield SDD Bootstrap)

## 概述

此命令用於將**既有專案（棕地專案）**引入規範驅動開發（SDD）工作流。與 `specify init` 從零開始不同，此命令會：

1. **逆向分析**現有程式碼庫，擷取架構 DNA
2. **產生客製化**的專案章程（constitution.md），反映專案實際原則
3. **適配範本**至專案特定的技術棧和程式碼風格
4. **驗證就緒狀態**確保 SDD 工作流可立即啟用

## ⚠️ 前置條件檢查

> **🚨 關鍵：此命令需要先初始化 spec-kit！**

**執行此命令前，請驗證以下內容：**

```bash
# 檢查 1：.specify/ 目錄存在（由 `specify init` 建立）
test -d .specify && echo "✅ .specify/ 存在" || echo "❌ 缺少 .specify/ - 請先執行 'specify init --here --ai claude --force'"

# 檢查 2：核心命令存在（來自 spec-kit 範本）
test -f .claude/commands/speckit.specify.md && echo "✅ 核心命令存在" || echo "❌ 缺少核心命令 - 請先執行 'specify init --here'"
```

**如果檢查失敗，必須先執行：**

```bash
# 先在目前目錄初始化 spec-kit
specify init --here --ai claude --force
```

**為什麼需要這樣做：**
- 此擴充功能**客製化**現有的 spec-kit 設定
- 核心命令（`/speckit.specify`、`/speckit.plan`、`/speckit.tasks`）來自 `specify init`，而非此擴充功能
- 此命令將用專案特定的版本**覆寫** constitution.md 和範本

## ⚠️ 關鍵：目標目錄結構

> **🚨 強制約束 - 必讀**
>
> 本提示詞產生的所有檔案**必須且只能**使用以下目錄結構。
> **任何偏離都將導致 spec-kit 工具鏈無法運作**。
> 在產生任何檔案前，請反覆確認路徑正確性。

**必須嚴格遵循以下目錄結構**，這是 spec-kit 工具鏈正常運作的前提：

```
[專案根目錄]/
├── .specify/                          # SDD 設定目錄（工具鏈依賴此路徑）
│   ├── memory/
│   │   └── constitution.md            # 專案章程（核心檔案）
│   ├── templates/
│   │   ├── spec-template.md           # 功能規範範本
│   │   ├── plan-template.md           # 實施計畫範本
│   │   └── tasks-template.md          # 任務清單範本
│   └── scripts/                       # 工作流腳本（可選，完整安裝時包含）
│       ├── bash/
│       └── pwsh/
└── specs/                             # 功能規範工作目錄
    └── [###-feature-name]/            # 按功能分支組織
        ├── spec.md
        ├── plan.md
        └── tasks.md
```

**重要說明**：
- `.specify/` 目錄名**必須**以點開頭，不是 `specify/`
- 範本檔案**必須**放在 `.specify/templates/` 下，不是根目錄的 `templates/`
- 章程檔案**必須**放在 `.specify/memory/constitution.md`
- `specs/` 目錄在根目錄下，用於存放每個功能的規範檔案

### 🚨 輸出前強制自檢清單

**在輸出任何檔案前，必須逐條確認以下檢查項全部通過：**

| # | 檢查項 | 正確 ✓ | 錯誤 ✗ | 驗證方法 |
|---|--------|--------|--------|----------|
| 1 | 設定目錄名稱 | `.specify/` | `specify/` | 必須以點開頭 |
| 2 | 章程檔案路徑 | `.specify/memory/constitution.md` | `memory/constitution.md` | 必須在 .specify 下 |

| 5 | 範本檔案完整 | 3 個範本檔案全部產生 | 遺漏任一範本 | spec/plan/tasks |

**❌ 如果任一檢查項失敗，必須立即修正後再輸出**

## 執行流程

### 階段 0: 專案探索 (Project Discovery)

**目標**: 系統性掃描專案，擷取核心特徵

#### 步驟 0.1: 結構掃描

首先，執行全面的專案結構掃描：

1. **識別專案根目錄標識檔案**：
    - 套件管理器: `package.json`, `pom.xml`, `build.gradle`, `Cargo.toml`, `pyproject.toml`, `go.mod`, `Gemfile`, `composer.json`
    - 設定檔: `.editorconfig`, `.prettierrc`, `.eslintrc.*`, `tsconfig.json`, `tox.ini`, `setup.cfg`
    - 版本控制: `.git/`, `.gitignore`, `.gitattributes`
    - CI/CD: `.github/workflows/`, `.gitlab-ci.yml`, `Jenkinsfile`, `azure-pipelines.yml`
    - 容器化: `Dockerfile`, `docker-compose.yml`, `kubernetes/`

2. **掃描目錄結構**：
   ```bash
   # 取得專案結構（深度 3 層，排除常見忽略目錄）
   find . -maxdepth 3 -type d \
     ! -path '*/node_modules/*' \
     ! -path '*/.git/*' \
     ! -path '*/vendor/*' \
     ! -path '*/__pycache__/*' \
     ! -path '*/target/*' \
     ! -path '*/build/*' \
     ! -path '*/dist/*' \
     -print
   ```

3. **統計程式碼分佈**：
    - 識別主要程式語言（按檔案數量和程式碼行數）
    - 識別測試目錄位置和測試框架
    - 識別文件位置（docs/, README.*, CONTRIBUTING.md 等）

#### 步驟 0.2: 技術棧擷取

基於掃描結果，建構技術棧概貌：

```markdown
## 技術棧概貌

**主要語言**: [識別的主要語言及版本]
**框架/執行環境**: [識別的主要框架]
**套件管理器**: [識別的套件管理工具]
**建構工具**: [識別的建構系統]
**測試框架**: [識別的測試工具]
**資料儲存**: [識別的資料庫/快取]
**部署平台**: [識別的部署目標]
**CI/CD 系統**: [識別的 CI/CD 工具]
```

#### 步驟 0.3: 架構模式識別

分析程式碼結構以識別架構模式：

1. **目錄命名模式分析**：
    - `src/controllers/`, `src/handlers/` → MVC/分層架構
    - `src/domain/`, `src/application/`, `src/infrastructure/` → 洋蔥/六角形架構
    - `services/`, `microservices/` → 微服務架構
    - `modules/`, `packages/` → 模組化單體
    - `frontend/`, `backend/` → 前後端分離

2. **程式碼組織模式**：
    - 按功能組織（feature-based）
    - 按層組織（layer-based）
    - 按領域組織（domain-based）

3. **常見模式偵測**：
    - Repository Pattern: `*Repository.*`, `*Repo.*`
    - Service Layer: `*Service.*`, `*Manager.*`
    - Factory Pattern: `*Factory.*`
    - DTO/VO: `*DTO.*`, `*VO.*`, `*Model.*`

#### 步驟 0.4: 程式碼規範擷取

從現有程式碼中擷取實際遵循的規範：

1. **從設定檔擷取**：
    - ESLint/Prettier 設定 → JavaScript/TypeScript 規範
    - .editorconfig → 通用格式規範
    - pylint/flake8/black 設定 → Python 規範
    - checkstyle/spotless 設定 → Java 規範

2. **從程式碼樣本推斷**：
    - 命名慣例（camelCase, snake_case, PascalCase）
    - 檔案命名模式（kebab-case.ts, PascalCase.java）
    - 註解風格（JSDoc, docstring, Javadoc）
    - 匯入組織方式

3. **從 Git 歷史推斷**（可選）：
    - 提交訊息格式
    - 分支命名策略
    - PR/MR 範本

#### 步驟 0.5: 多模組/多子專案分析 ⚠️ 關鍵

**對於包含多個子模組的專案（如 Java Maven/Gradle 多模組、Node.js Monorepo 等），必須執行此步驟**。

##### 1. 識別多模組專案類型

| 專案類型 | 識別特徵 | 識別方法 |
|----------|----------|----------|
| **Maven 多模組** | 根目錄 `pom.xml` 包含 `<modules>` | 解析 `pom.xml` 中的 `<modules>` 標籤 |
| **Gradle 多模組** | 根目錄 `settings.gradle(.kts)` 包含 `include` | 解析 `include` 語句 |
| **Node.js Monorepo** | 根目錄 `package.json` 包含 `workspaces` | 解析 `workspaces` 設定 |
| **Go Workspace** | 存在 `go.work` 檔案 | 解析 `use` 指令 |
| **Rust Workspace** | `Cargo.toml` 包含 `[workspace]` | 解析 `members` 設定 |
| **Python Monorepo** | 多個 `pyproject.toml` 或 `setup.py` | 搜尋多個套件定義檔 |

##### 2. 列舉所有子模組

**必須掃描並列出專案中的所有子模組**：

```markdown
## 子模組清單

| 模組名 | 路徑 | 模組類型 | 打包方式 |
|--------|------|----------|----------|
| [module-1] | [path/to/module-1] | [jar/war/pom/lib] | [artifact 類型] |
| [module-2] | [path/to/module-2] | [jar/war/pom/lib] | [artifact 類型] |
| ... | ... | ... | ... |
```

**Java Maven 專案範例**：
```bash
# 解析父 pom.xml 取得所有子模組
grep -oP '(?<=<module>)[^<]+' pom.xml
```

**Java Gradle 專案範例**：
```bash
# 解析 settings.gradle 取得所有子模組
grep -oP "include\s*['\"]([^'\"]+)['\"]" settings.gradle
```

##### 3. 分析子模組職責

**對每個子模組進行職責分析和分類**：

| 模組分類 | 典型命名模式 | 職責描述 |
|----------|--------------|----------|
| **API/介面層** | `*-api`, `*-interface`, `*-facade` | 對外公開的介面定義，DTO/VO |
| **服務實作層** | `*-service`, `*-impl`, `*-core` | 業務邏輯實作 |
| **資料存取層** | `*-dao`, `*-repository`, `*-persistence` | 資料庫操作，Entity 定義 |
| **通用工具層** | `*-common`, `*-utils`, `*-base` | 共用工具類別，基礎設施 |
| **Web/控制器層** | `*-web`, `*-controller`, `*-rest` | HTTP 端點，請求處理 |
| **啟動/引導模組** | `*-app`, `*-boot`, `*-starter` | 應用程式啟動入口，設定彙整 |
| **領域模組** | `*-domain`, `*-model` | 領域模型，業務實體 |
| **整合/轉接器** | `*-integration`, `*-adapter`, `*-client` | 第三方系統整合 |

**產生模組職責對映表**：

```markdown
## 模組職責對映

| 模組 | 職責分類 | 核心職責 | 可放置的程式碼類型 |
|------|----------|----------|------------------|
| user-api | API/介面層 | 使用者服務介面定義 | DTO, VO, 介面定義, 列舉 |
| user-service | 服務實作層 | 使用者業務邏輯 | Service 實作, 業務驗證, 交易處理 |
| user-dao | 資料存取層 | 使用者資料持久化 | Entity, Repository, Mapper |
| common | 通用工具層 | 共用基礎設施 | 工具類別, 常數, 例外定義 |
```

##### 4. 分析模組間相依關係

**必須分析並記錄模組間的相依關係**：

**Maven 專案**：
```bash
# 分析每個模組的相依性
for module in $(grep -oP '(?<=<module>)[^<]+' pom.xml); do
  echo "=== $module ==="
  grep -A1 '<dependency>' $module/pom.xml | grep -oP '(?<=<artifactId>)[^<]+'
done
```

**產生相依關係圖**：

```markdown
## 模組相依關係

```
[app-boot]
├── [user-service]
│   ├── [user-api]
│   ├── [user-dao]
│   └── [common]
├── [order-service]
│   ├── [order-api]
│   ├── [order-dao]
│   ├── [user-api]  ← 跨模組相依
│   └── [common]
└── [common]
```

### 相依規則

- **允許的相依方向**: 上層 → 下層（service → api → common）
- **禁止的相依方向**: 下層 → 上層、循環相依
- **跨領域相依**: 僅透過 API 模組相依，不直接相依 Service 實作
```

##### 5. 確定新功能程式碼放置決策樹

**基於模組分析，建立程式碼放置決策指南**：

```markdown
## 程式碼放置決策樹

當需要新增功能時，按以下決策樹確定程式碼位置：

1. **這是對外公開的介面/DTO 嗎？**
   - 是 → 放入對應領域的 `*-api` 模組
   - 否 → 繼續

2. **這是業務邏輯/服務實作嗎？**
   - 是 → 放入對應領域的 `*-service` 模組
   - 否 → 繼續

3. **這是資料庫實體/資料存取嗎？**
   - 是 → 放入對應領域的 `*-dao` 模組
   - 否 → 繼續

4. **這是跨多個領域共用的工具嗎？**
   - 是 → 放入 `common` 模組
   - 否 → 繼續

5. **這是新的業務領域嗎？**
   - 是 → 考慮建立新的模組群組（xxx-api, xxx-service, xxx-dao）
   - 否 → 諮詢架構師

### 模組選擇速查表

| 程式碼類型 | 目標模組 | 範例 |
|----------|----------|------|
| 新 REST API | `[領域]-web` 或 `[領域]-controller` | UserController |
| 新 Service 介面 | `[領域]-api` | IUserService |
| Service 實作 | `[領域]-service` | UserServiceImpl |
| 資料庫 Entity | `[領域]-dao` | UserEntity |
| 資料庫操作 | `[領域]-dao` | UserRepository |
| DTO/VO | `[領域]-api` | UserDTO |
| 工具類別 | `common` | StringUtils |
| 例外定義 | `common` 或 `[領域]-api` | UserNotFoundException |
| 設定類別 | `[領域]-boot` 或 `app` | DataSourceConfig |
```

mkdir -p .specify/memory

#### 步驟 1.2: 原則推導

基於探索報告，推導專案應遵循的原則：

**原則推導邏輯**：

| 探索特徵 | 推導原則 | 原則描述 |
|----------|----------|----------|
| 存在獨立模組/套件結構 | 模組化原則 | 功能必須作為獨立模組實作，保持清晰邊界 |
| 存在統一測試目錄 | 測試優先原則 | 所有功能必須撰寫測試，測試先於實作 |
| 存在 API 目錄/路由 | 介面契約原則 | 所有對外介面必須先定義契約 |

| 存在統一錯誤處理 | 錯誤處理原則 | 錯誤必須透過統一機制處理 |
| 存在日誌設定 | 可觀測性原則 | 所有操作必須產生可追蹤的日誌 |
| 存在 CI/CD 設定 | 持續整合原則 | 所有變更必須通過自動化驗證 |

建立 `.specify/memory/constitution.md`：

**相依規則**: [相依管理規則]

### VI. 程式碼複用與理解原則

#### 禁止程式碼重複

**強制規則**: 建立新的類別和方法之前，**必須**先查找現有的類別和方法。

**執行步驟**:
1. 搜尋專案中是否已有相似功能的類別/方法
2. 檢查 component/common/utils/bll/service 等共用模組
3. 確認無可複用程式碼後，才建立新程式碼

**驗證方法**: 程式碼審查時檢查是否存在重複實作

#### 理解現有程式碼

**工具**: 當需要理解現有程式碼時，使用 Repomix 為專案產生/更新 skills。

```bash
# 產生/更新專案技能文件
repomix --skill-generate
```

**使用時機**:
- 首次接觸專案時
- 大型功能開發或重構前

## 🧪 TDD 測試驅動開發紀律 *(強制)*

### TDD 核心原則

**所有新功能開發必須嚴格遵循 TDD 紅-綠-重構循環**：

```
┌─────────────────────────────────────────────────────────────┐
│                    TDD 開發循環（強制）                       │
├─────────────────────────────────────────────────────────────┤
│  1. 🔴 RED    │ 先寫失敗的測試 → 執行測試 → 確認失敗       │
│  2. 🟢 GREEN  │ 寫最少程式碼讓測試通過 → 執行測試 → 確認通過 │
│  3. 🔵 REFACTOR │ 重構程式碼 → 執行測試 → 確認仍然通過       │
│  4. 🔁 REPEAT │ 返回步驟 1，處理下一個測試案例              │
└─────────────────────────────────────────────────────────────┘
```

### TDD 執行規則

| 規則 | 描述 | 違反後果 |
|------|------|----------|
| **先測試後實作** | 任何業務程式碼撰寫前，必須先存在對應的失敗測試 | 程式碼審查不通過 |
| **最小實作** | 每次只寫讓目前測試通過的最少程式碼 | 避免過度設計 |
| **測試即文件** | 測試案例命名必須清晰描述業務情境 | 可維護性差 |
| **100% 新程式碼覆蓋** | 所有新增業務邏輯必須有對應測試 | 不可合併 |

### TDD 在 SDD 工作流中的位置

```
規範(Spec) → 計畫(Plan) → 任務(Tasks) → [TDD 循環] → 完成
                                          │
                    ┌─────────────────────┴─────────────────┐
                    │  每個任務內部:                          │
                    │  T1: 撰寫測試案例（🔴 預期失敗）        │
                    │  T2: 執行測試確認失敗                   │
                    │  T3: 實作功能程式碼（🟢 最小實作）       │
                    │  T4: 執行測試確認通過                   │
                    │  T5: 重構最佳化（🔵 保持測試通過）       │
                    └───────────────────────────────────────┘
```

### TDD 檢查點 *(每個使用者故事實作後必須驗證)*

## 通用指南

### 棕地專案特殊考量

1. **尊重現有架構**: 不要試圖「改進」現有架構，而是準確反映它
2. **技術棧鎖定**: 棕地專案的技術棧是既定事實，章程必須錨定它
3. **漸進式引入**: SDD 工作流應無縫融入現有開發流程
4. **相容性優先**: 所有產生的程式碼必須與現有程式碼風格一致
5. **禁止程式碼重複**: 建立新的類別和方法之前，必須先查找現有的類別和方法，如果沒有可複用的程式碼，才建立新程式碼
6. **理解現有程式碼**: 當需要理解現有程式碼時，使用 Repomix 為專案產生/更新 skills，命令：`repomix --skill-generate`

### 路徑引用規範

**在範本檔案中引用其他檔案時**：

| 引用對象 | 正確路徑 | 錯誤路徑 |
|----------|----------|----------|
| 專案章程 | `.specify/memory/constitution.md` | `memory/constitution.md` |
| 規範範本 | `.specify/templates/spec-template.md` | `templates/spec-template.md` |
| 功能規範 | `specs/[###-feature]/spec.md` | `.specify/specs/[###-feature]/spec.md` |

### 處理不確定性

當無法從程式碼中確定某些慣例時：

1. **優先使用業界標準**: 基於技術棧的最佳實務
2. **標記待確認**: 使用 `[NEEDS CLARIFICATION: 問題描述]` 標記
3. **限制數量**: 最多 3 個待確認項
4. **提供選項**: 為每個待確認項提供 2-3 個建議選項

### 錯誤處理

| 情境 | 處理方式 |
|------|----------|
| 無法識別專案類型 | 詢問使用者確認主要技術棧 |
| 目錄結構異常 | 使用標準結構但標記為「非標準專案」 |
| 無測試程式碼 | 章程中新增「建立測試實務」原則 |
| 多語言專案 | 識別主要語言，章程中列出所有語言 |
